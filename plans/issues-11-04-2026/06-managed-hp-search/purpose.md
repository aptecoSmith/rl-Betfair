# 06 — Managed Hyperparameter Search

## Problem

Each training run initialises its population with purely random
hyperparameters — no awareness of what's been tried before. Over
many runs we may repeatedly explore the same regions while leaving
others untouched. There's no map of "what we've tried" and "what
worked", and no strategy for systematically covering the search
space.

## Vision

Build a two-phase exploration strategy:

**Phase 1 — Global coverage pass:** Systematically sample across the
full hyperparameter space so that over N training runs, we build a
coverage map with no large gaps. Each run starts from a deliberately
new region. Within each run, the genetic algorithm does local
exploration (crossover + mutation) as it does today.

**Phase 2 — Focused exploitation:** Once the global pass gives us a
map of which regions are promising, do targeted runs that zoom into
the best regions with tighter ranges.

Meanwhile, every run produces trained models and descendants — so
even during the coverage pass, we accumulate usable models.

## What already exists (unused)

`training/training_plan.py` already has:

- **`compute_coverage()`** (lines 479-571) — analyses historical
  models from the registry, buckets numeric genes into N bins,
  flags undercovered architectures (< 15 samples) and poorly
  covered genes (< 60% of buckets non-empty). Returns a
  `CoverageReport`.

- **`bias_sampler()`** (lines 574-654) — given a `CoverageReport`,
  returns `BiasedSpec` wrappers that weight empty buckets at 1.5x.
  `sample_with_bias()` picks a bucket probabilistically, then
  uniform-samples within it.

- **Coverage API endpoint** — `GET /api/training-plans/coverage`
  computes and returns coverage analysis.

**None of this is wired into `initialise_population()`.** The initial
population is always purely random uniform.

## Design

### 1. Exploration log (new DB table)

New table: `exploration_runs`

```sql
CREATE TABLE exploration_runs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL,           -- links to training plan/run
    created_at TEXT NOT NULL,
    seed_point TEXT NOT NULL,       -- JSON: the seed hyperparameters
    region_id TEXT,                 -- label for the region (e.g. "point_3_of_10")
    strategy TEXT NOT NULL,         -- "grid", "sobol", "coverage_biased", "manual"
    coverage_before TEXT,           -- JSON: CoverageReport at time of seeding
    notes TEXT
);
```

This gives us a historical record of where we've seeded training
runs, separate from the individual model records.

### 2. Systematic seed point generation

When starting a new training run, instead of random sampling, offer
a systematic strategy:

**Sobol / Latin Hypercube sampling** across the full search space:
- User chooses N points to explore (e.g. 10).
- System generates N well-spaced seed points using Sobol sequences
  (quasi-random, low-discrepancy — better coverage than pure random).
- Each training run takes the next unused seed point.
- The seed point defines the *centre* of the initial population.
  Individual models in the population are scattered around it with
  small perturbations.

**Coverage-biased sampling** (uses existing infrastructure):
- Compute `CoverageReport` from all historical models.
- Use `bias_sampler()` to generate seed points that favour
  unexplored regions.
- This is adaptive: each run automatically targets gaps left by
  previous runs.

### 3. Seed point → initial population

The seed point is a single hyperparameter vector. The population of
N models is generated by perturbing around it:

```python
for i in range(population_size):
    hp = {}
    for gene in seed_point:
        # Small perturbation: ±10% of range around seed value
        hp[gene] = perturb(seed_point[gene], spec, sigma=0.1)
    agents.append(hp)
```

This keeps the initial population clustered in one region of the
space, so the genetic algorithm can do meaningful local search
rather than starting from noise.

### 4. Training run workflow

1. **Plan creation:** User selects exploration strategy:
   - "Next Sobol point" — auto-picks the next point from a
     pre-generated sequence.
   - "Coverage-biased" — auto-picks based on gap analysis.
   - "Manual" — user specifies seed hyperparameters.
   - "Random" — current behaviour (backward compat).

2. **Seed generation:** System generates the seed point and records
   it in `exploration_runs`.

3. **Population init:** `initialise_population()` uses the seed
   point + perturbation instead of random sampling.

4. **Training:** Genetic algorithm runs as normal (crossover,
   mutation, selection) — local search around the seed region.

5. **Post-run:** Coverage map is updated (all new models are in the
   registry). Next run can query it.

### 5. Coverage dashboard

A new UI page or section showing:
- Heat map of explored vs unexplored regions per gene.
- Which seed points have been tried and their outcomes.
- Suggested next seed point based on coverage gaps.
- Per-architecture exploration depth.

### 6. Phase 2: focused exploitation

Once the global pass is done (operator decides when), switch to:
- Tighter search ranges around the best-performing regions.
- Training plans auto-generated from the top-scoring models' gene
  neighbourhoods.
- Higher mutation rates within the focused region for fine-tuning.

This is a future extension — the infrastructure from Phase 1 (seed
tracking, coverage analysis) enables it naturally.
