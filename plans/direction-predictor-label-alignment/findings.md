# Findings — direction predictor / label alignment

## Original 2026-05-24 (morning) — what we THOUGHT we found

The `tools/direction_head_inspection.py` brute-force scan reported
that ALL 12 direction-predictor obs columns had |rho| < 0.05 with the
offline `direction_label_scan` labels. Combined with the
`extract_labels_prototype.py` finding that the betfair-predictors
model was trained on 60s/180s/420s clock-time horizons while our
labels use a 60-TICK horizon, the diagnosis was: predictor and labels
speak different temporal languages, hence ~zero correlation.

We then added a time-endpoint label mode to `direction_label_scan.py`
(420s horizon, signed-tick semantics matching the predictor), scanned
one day, and re-ran the brute-force scan. **The correlation was STILL
~0.04 — no meaningful improvement.**

That should have been a red flag: a label-semantics fix that does
nothing strongly suggests the diagnosis is wrong.

## 2026-05-24 (afternoon) — the real finding

Direct inspection of the oracle cache obs vector revealed that the
direction-predictor columns
(`dir_q10_1m / dir_q50_1m / dir_q90_1m / dir_q10_3m / ... /
dir_fire_drift / dir_fire_shorten / dir_fire_no_signal`)
are **all zero, every sample, every day**:

```
dir_q10_1m   : min=0.000  mean=0.000  std=0.000  max=0.000
dir_q50_1m   : min=0.000  mean=0.000  std=0.000  max=0.000
...all 12 identical: all-zero across 82,754 samples on 2026-04-11.
```

Comparing to the actual training-time env (built fresh with
`--use-direction-predictor` + predictor bundle, walked 15 ticks):

```
dir_q10_1m   :  −0.79   (small signed ticks, sensible)
dir_q50_1m   :  −0.02
dir_q10_3m   :  −2.47
dir_q10_7m   :  −3.90
dir_fire_drift   : 0.0   (proper bool)
dir_fire_shorten : 0.0
dir_fire_no_signal: 1.0
```

**The oracle cache obs is generated by `training_v2.oracle_cli scan`,
which builds the env WITHOUT `--use-direction-predictor` (no
predictor bundle threaded through).** So the 12 direction columns are
zero-filled placeholders. My `direction_signal_probe.py` and
`direction_head_inspection.py` were correlating ZEROS against labels.

## Reframe: what we actually know and don't know

**Confirmed:**
* The pretrained `betfair-predictors` direction model (V4) outputs
  signed-tick forecasts at 3m/7m/15m horizons.
* The env's `_compute_tick_predictor_outputs` correctly populates
  the 12 obs slots when the env is built with
  `use_direction_predictor=True` + a predictor bundle.
* The cohort's training-time obs DOES include those values
  (verified by direct env walk).
* The head's BCE doesn't descend during cohort training (16 days
  flat at ~1.14, the pos-weighted uniform-0.5 floor at 18% positive
  rate).

**Still unknown (the original question):**
* Whether the predictor columns DO correlate with the offline labels
  when measured properly on training-time obs.
* Whether the head failing to descend BCE is due to:
  (a) the predictor signal being uninformative for the labels;
  (b) the head architecture being too weak;
  (c) PPO gradient flow dominating and starving the supervised head;
  (d) something else.

**Tentatively useful:**
* The new `v2_time_endpoint_signed_tick` label mode added today
  matches the predictor's training-label semantics, so even if the
  predictor's outputs DO correlate with the original v1 labels, the
  v2 labels should be a strict improvement. Keep the new code.

## Next step — proper probe

Two equivalent options:

**(A) Add `--use-direction-predictor` to `oracle_cli`** so the next
oracle re-scan includes the predictor columns. Then re-run
`direction_signal_probe.py` and `direction_head_inspection.py`
brute-force scan. This is the cleanest because it reuses existing
diagnostics, and the better-instrumented oracle cache helps every
future investigation.

**(B) Write a one-off probe** that builds env with predictor in
memory, walks one day, and emits correlation tables on the fly.
Faster, throwaway. Doesn't unblock future diagnostics.

Picking (A). Estimated 30 min for cache fix + re-scan, then probe
results land.

## Why the previous probe still saw "linear classifier descends 10%
relative below uniform floor"

Even with zero predictor columns, the obs vector still has 11 OTHER
lean keys per runner (price/spread/velocity/champion/ranker) plus
the global market + velocity + agent state. A logistic regression on
those 11+ × 14 runners' worth of features can pick up SOME signal
about future price movements purely from price velocities, spread
dynamics, and the champion's outright p_win. That's where the 10-12%
descent came from.

The descent is HONEST signal in obs — but it ISN'T from the
direction predictor. So the diagnosis "signal is in obs, head must
be undertrained at loss_weight=0.05" was right in spirit but for the
wrong reason. The predictor's contribution is still TBD.

## 2026-05-24 (afternoon, part 2) — final reframe

Added `--use-direction-predictor` to `training_v2.oracle_cli scan`
and re-scanned 2026-04-11. Re-ran the brute-force scan against the
populated cache, **vs the ORIGINAL v1 (tick-crossing) labels**:

```
=== v1_tick60 labels (N=82754, back+=18.5%, lay+=18.2%) ===
  col                    vs label_back  vs label_lay
  dir_q90_3m                   -0.1350       -0.0474  ***
  dir_q90_7m                   -0.1323       -0.0488  ***
  dir_q90_1m                   -0.1221       -0.0400  ***
  dir_q50_7m                   -0.1102       -0.0407  ***
  ...
```

Versus v2 endpoint labels — almost all |rho| < 0.022. v2 endpoint
labels are **worse** than v1 tick-window labels at this task.

Linear probe (sklearn LogisticRegression with class_weight='balanced'
on the full 574-dim obs + v1 labels):

```
direction_back: uniform floor 1.130 → trained val BCE 0.949 → 16.0% descent
direction_lay:  uniform floor 1.134 → trained val BCE 0.911 → 19.7% descent
```

Both descents are well above the 5%/20% acceptance gates in
`hard_constraints.md §8`. **The signal IS extractable from obs, AND
the predictor's contribution is meaningful, AND the v1 labels are
the right target.**

## Real root cause (still unconfirmed)

The agent's policy `direction_prob_head` failed to descend BCE
during the Phase-15 cohort (16 days of training at gene_loss_weight
= 0.515 → BCE stayed at ~1.14 ≈ uniform floor) despite the signal
being demonstrably present and extractable in obs.

Possible reasons (in order of suspicion):

1. **Backbone mediation.** The head reads `backbone(obs)`, not raw
   obs. PPO reshapes the backbone toward policy/value-relevant
   features, potentially destroying the direction signal before the
   head sees it.
2. **Loss weight still too small in practice.** Even at 0.515, the
   supervised gradient may be dwarfed by the value + policy
   gradients into the shared backbone. Bumping to 5.0+ might fix.
3. **Head architecture.** A single `nn.Linear` from hidden_size to
   max_runners may not have the capacity to extract the signal that
   a 574-dim logreg can.
4. **PPO/BCE gradient interaction.** The two loss objectives may
   pull the head in incompatible directions.

## Validated and rejected fixes

* **REJECTED:** label semantics fix (v2 endpoint time-horizon mode).
  Code was added but the labels turn out to correlate WORSE than v1.
  Keep the code (it's harmless and other future work might use it)
  but DO NOT use these labels in the cohort.

* **VALIDATED:** `--use-direction-predictor` flag on
  `training_v2.oracle_cli scan`. This was a real bug — the oracle
  cache was generated without the predictor, so the 12 predictor
  obs columns were zero-filled. The flag fixes this and unblocks
  all future predictor diagnostics. KEEP.

## Next step — probe the backbone

Smallest informative test: load agent 1's saved weights, walk a few
hundred obs through the backbone (LSTM), capture hidden state, train
a logreg on `(hidden_state → v1 label)`. Compare descent to the
full-obs logreg.

If backbone hidden state achieves similar 16-20% descent → backbone
preserves signal → fix is to bump direction_prob_loss_weight gene
range upward and/or widen the head.

If backbone hidden state descents much less than full-obs logreg →
backbone destroys signal → deeper fix needed (e.g. pass obs[dir_*]
directly into actor_head bypassing backbone).

## 2026-05-24 (evening) — final reframe (commit 2bfe4cc)

The backbone probe (above) showed lstm_last has ~0% direction signal.
That LED me to propose a "residual obs path into direction_prob_head"
fix. While implementing it, I noticed the head's existing forward
code at agents_v2/discrete_policy.py:704-713 ALREADY bypasses
lstm_last and reads the per-runner obs feature slice directly.

So why doesn't it learn? Loading agent 1's trained weights and
inspecting:

```
Head architecture:
  0.weight (LayerNorm): shape=(143,)
  0.bias:               shape=(143,)
  1.weight (Linear):    shape=(64, 143)
  ...
```

**The head was sized for RUNNER_DIM=143 (full obs).** But the cohort
runs with `--predictor-lean-obs`, which means env's per-runner block
is 23 dims, not 143.

The policy's runner-block extractor (lines 487-493 before fix) has a
"test-mode fallback" that anchors at obs offset 0 (instead of
MARKET_DIM+VELOCITY_DIM = 48) and takes whatever fits (574 for lean
obs), padding with zeros up to 14*143 = 2002. Result: the head's
"per-runner feature slice" for runner 0 = obs[0:143] = market state
+ velocity + runner 0's 23 actual features + part of runner 1's
features. Runners 8+ see mostly zeros. Structurally garbage input.

The head was trained for 16 days on garbage. BCE flat at the random
floor because gradient updates on garbage input go nowhere useful.

### The real fix (commit 2bfe4cc)

* Surface `env.active_runner_dim` publicly.
* Thread it into `DiscreteLSTMPolicy(runner_dim=…)`.
* Use it instead of the module-level `RUNNER_DIM` constant in 5
  places (LayerNorm dim, first Linear dim, `_runner_block_size`
  computation, two view/reshape calls).
* 7-test regression guard in `tests/test_v2_direction_head_runner_dim.py`.

### Why all the false starts mattered

Each false start eliminated a possibility:

* "Pos-weighted floor not 0.693" → caught a measurement
  misinterpretation, not a code bug
* "Labels misaligned with predictor horizon" → led to a usable
  v2 endpoint label mode (kept as inert infrastructure) but
  wasn't the fix
* "Oracle cache had zero predictor columns" → caught a real bug
  in oracle_cli that would otherwise have broken every future
  predictor diagnostic. Fixed.
* "Backbone destroys signal" → true for lstm_last but
  irrelevant; the head doesn't read lstm_last. The probe tool
  is still valuable; the diagnosis pointed at the wrong fix.

The actual cause was only findable by inspecting the trained
head's weight SHAPES, which I did almost by accident while
implementing the residual-obs fix proposal.

### Expected post-fix behaviour

Per the per-runner-obs logreg numbers (back: +5.7%, lay: +12.3%
BCE descent), the head — a 2-layer MLP with more capacity than
the logreg — should match or exceed those descents within a few
days of training. The Phase-15 cohort relaunched on top of commit
2bfe4cc should show `dir_bce_back` and `dir_bce_lay` descending
below 1.10 within ~5 days.
