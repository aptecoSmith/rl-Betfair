# AGENTS.md — operating rules for rl-betfair's in-repo knowledge wiki

You are maintaining **rl-betfair's knowledge wiki** — an in-repo copy of the operator's `llm-wiki-v3`
engine (an Engelbart/CoDIAK spine: granular IDs, claim-level provenance, typed links, an embedded
queryable projection, a captured dialog record). Read this file first, every session. The
deterministic engine is `scripts/wiki_tool.py` — call it; don't reimplement it. Provenance and exactly
what was copied/dropped from upstream: [`VENDORED_FROM.md`](VENDORED_FROM.md).

> **What this wiki is for.** It is the **deep, pull-on-demand layer** for rl-betfair's accreted
> knowledge — the topic-keyed synthesis (claims + provenance, supersession chains, the decision/dialog
> record) deliberately *kept out* of the always-loaded `CLAUDE.md` + `memory/`. A session pulls what it
> needs (the CLI here, or the read-only MCP server registered in `../.mcp.json`) instead of
> carrying it all in resident context. Design rationale: `../plans/memory-improvements/`. The **binding
> invariants** at the bottom of this file govern the engine — do not violate them.

## Session start — the opener
When the user greets you or sends an **opener trigger** (`open`, `start`, `start wiki`, `go`, `wiki`)
at the beginning of a session, run the opener and reply with a short status block — do not wait to be
asked, and do not ingest or edit anything yet:

1. Read this file and the active cloud's `index.md` (`shared/index.md`).
2. Get counts: `python scripts/wiki_tool.py doctor` and `python scripts/wiki_tool.py ingest-log --summary`.
3. Check for pending work in `inbox/pending/` (the `Files/` drop and `urls.md`), `inbox/static/`
   (append-only notes, snapshotted not moved), and the root `Clippings/` folder (web-clipper drops,
   ingested and moved like the inbox).
4. Reply with the **opener**:
   - active cloud + note/source counts,
   - anything pending to ingest (dropped files, unscanned URLs), or "nothing pending",
   - the line: **"What would you like to ingest or ask?"**

Keep it to a few lines. Wait for the user to confirm before ingesting or editing notes.

## The model in brief
- **One cloud: `shared/`.** rl-betfair knowledge is all "work" — everything lives in `shared/`
  (`personal/` is unused here). The engine still supports a `personal/` cloud; this wiki simply doesn't
  populate one, and a `shared/` note must never depend on `personal/`.
- **Sources are referenced, not copied.** Register location + hash + title in `Schema/sources.jsonl`;
  never copy source bytes into the repo. For the repo-scan path the source IS the in-repo file (e.g.
  `plans/.../findings.md`) — referenced in place. Dropped inbox files are the one staging exception;
  `inbox/static/` notes are referenced in place and snapshotted, never moved.
- **The graph is the product.** Every note links into a hub (`shared/index.md`, auto-generated by
  `build`); isolated notes and unreachable clusters are failures, not cosmetic.

## Ingestion — two entry points
rl-betfair knowledge enters the wiki two ways:

1. **Repo-scan (bulk — the rl-betfair corpus).** `python scripts/scan_repo.py` walks the rl-betfair
   tree (`plans/**/*.md`, `docs/**/*.md`, and root knowledge docs: `CLAUDE.md`, `genes_census.md`,
   `plans/EXPERIMENTS.md`, `plans/EXPLORATIONS.md`), **registers each as a source** (referenced in
   place — the files stay in the repo, never copied), and queues them into a `batch` ledger named
   **`repo-md`**. Work that queue with the **batch script** — which is `scripts/batch.py`, **NOT** a
   `wiki_tool` subcommand — one source at a time:
   ```
   python scripts/batch.py --name repo-md status        # progress + the next source
   python scripts/batch.py --name repo-md next          # claim one → ingest (extract/ingest skill)
   python scripts/wiki_tool.py finalize-ingest          # the gate (commits the chunk)
   python scripts/batch.py --name repo-md done <src-id> # after a note cites it
   ```
   Scope is markdown knowledge; ingesting `.py` code is a later option. `scan_repo.py` is rl-betfair's
   own thin wrapper over the engine's `register` + `batch` — see `python scripts/scan_repo.py --help`.
2. **Inbox drop (external ML docs you find).** The inbox is for **outside** material — an ML paper, a
   blog post, a vendor doc. Drop a file into `inbox/pending/Files/`, a URL into `inbox/pending/urls.md`,
   or a web-clip into `Clippings/`, then ingest via the `ingest` skill; `finalize-ingest` moves it to
   `processed/`. This is the engine as shipped.

Both paths produce the **same note shapes:** a `concept` note per mechanism (the matcher, equal-profit
sizing, naked variance, the held-out protocol), a `synthesis` note per cross-cohort lesson, a `project`
note per campaign, `claims` carrying the numbers + their source run, `dialog` entries for the decisions.

## Skills (repeatable workflows)
Each skill is a short playbook in `skills/<name>/SKILL.md`. **When a request matches one, open that
file and follow it** — don't improvise the workflow. These are **playbooks to READ** (`Read` the
SKILL.md, then run the `wiki_tool.py` / `batch.py` commands it lists) — **NOT** Claude Code skills to
invoke. Do **not** call the Skill tool on them (Claude Code may surface them from `skills/` but blocks
invocation; that's expected — just read and follow).

| Skill | Use it when… |
|---|---|
| `curate` | **before ingesting** — classify the source's doc-type and extract it the way that type needs (a plan findings doc ≠ a census table ≠ an experiments log); `Schema/doc-types.md` |
| `ingest` | turning one source into linked notes ("ingest X", or confirmed pending intake) |
| `extract` | the source is large/dense — the anti-shortcut extraction problem (a fat `plans/` writeup) |
| `query` | answering a question from the wiki, and filing worthwhile answers back |
| `maintain` | a routine health pass (rebuild indexes, refresh source presence, run the gates) |
| `audit` | a semantic editorial pass — contradictions, stale claims, gaps (on a cadence) |
| `dream` | "dream" / "dream for 30 min" — reflect on the graph, propose & auto-answer questions, queue research |
| `batch` | a large corpus to ingest over time, resumably (one source at a time) — the repo-scan queue |
| `static` | an append-only note in `inbox/static/` (a running log) |
| `gdrive-ingest` | ingesting a Google Drive folder |

If the user names a skill or uses one of these triggers, read the matching `SKILL.md` and run it.

## Ingesting a source (one source at a time)
1. **Register** the source (`wiki_tool.py register …`) → get a `src-id`.
2. **Read it fully.** Never skim or sample. For large/dense docs, use the extraction engine below.
3. **Discover the entities** the source names: `wiki_tool.py discover --source <src-id>` — the
   per-document worklist of people/orgs/tools/places/terms, split into "already in the wiki (link
   these)" vs "not represented (create a node, link one, or skip)". This is the *search the document*
   step; run it before writing notes so nothing named slips through.
4. **Search first:** `wiki_tool.py search "…"` across the active cloud before creating notes.
5. **Write breadth.** A new Entity for every named cohort/campaign/mechanism/tool; a new Concept for
   every distinct idea; Topics/Projects where warranted. Typical ingest touches many notes. Every named
   entity from `discover` must become a node, a `[[link]]`/alias in a citing note, OR an explicit
   `wiki_tool.py entity-skip --source <id> --term "…" --reason "…"` (the auditable "not node-worthy").
6. **Lift key quotes into the notes** so citations survive even if the source later moves.
7. **Cross-link** every new note to at least one hub and to related notes via `[[wiki-links]]`.
8. **Finalize:** `wiki_tool.py finalize-ingest`. The ingest is not done until this passes — it blocks on
   the **under-extraction**, **claimless**, AND **unrepresented-entity** coverage ERRORs.

## Static (append-only) sources
A note you keep reopening and appending to lives in `inbox/static/`. Unlike a `pending/` drop, a static
source is **referenced in place and never moved** — you keep editing the same file. To ingest one (see
the `static` skill):
1. **Register** it if new (`register --path inbox/static/<f>`); the src-id is stable across edits.
2. **Diff, don't re-read everything.** Compare it to the latest snapshot in `inbox/processed/static/`
   and ingest **only the newly-added content**. No prior snapshot ⇒ ingest it all.
3. **Finalize.** `finalize-ingest` copies a fresh dated snapshot to `inbox/processed/static/` (only
   when the file changed) and leaves your original in place for the next entry.

## Web clippings
The Obsidian Web Clipper saves articles to the root `Clippings/` folder. Treat each as a normal pending
drop: ingest it like a `Files/` drop, then `finalize-ingest` moves it to `inbox/processed/Clippings/`.

## The anti-shortcut extraction engine (large / dense docs)
Producing 5 notes from a 60-page campaign writeup is a failure. For such docs:
1. **Segment** by structure: doc → section → finding → claim (provenance kept).
2. **Process each segment independently** — enumerate *every* entity/concept/finding/decision in that
   segment before moving on. Don't read the whole doc then emit a handful of notes.
3. A segment that yields **zero** notes must be justified (genuinely contentless) or it's flagged.
4. **Dedup/merge** concepts that recur across segments — link, don't duplicate.
5. Expect to clear the **coverage floor**. `finalize` warns when you're under it — treat that warning
   as "what did I skip?"
Granularity: structural-segment by default; finer for reference-grade material.

## Querying
Read `index.md` → `search` → open top notes → answer from compiled notes with citations. Open a raw
source only if a note is a seed/incomplete or the user wants verification. **File worthwhile answers
back** as a `query` note so knowledge compounds.

## Tags & types
One structural **type** per note (`topic, concept, entity[+subtype], project, synthesis, log, query`)
plus controlled **context tags** (`work, home, research, meetings, lessons`). Unknown tags fail
`validate`. (rl-betfair knowledge is `work` / `research` / `lessons`; `home` is unused.)

## Hard rules
- Don't copy source bytes into the repo (register references instead).
- Don't invent citations. No source ⇒ find one or don't make the claim.
- Don't leave orphans; every note reaches a hub.
- Don't commit secrets or machine-local paths (the `audit_public` gate blocks them).
- When a rule itself is wrong (linking/ingest keeps going astray), fix the rule here and in the
  schema/tooling — not just the symptom.
- **"Process everything" / "ingest everything" means *finish the job*.** Don't mark heavy sources
  `batch done` with only a structural-overview skim and call the rest "TBD" — that's the exact shortcut
  the extraction engine forbids. Clear the coverage floor before closing the source. Don't ask which to
  prioritise — process all of them.

## Replication
Every `register` and `finalize-ingest` is recorded append-only in `Schema/ingest-log.jsonl` (the
replication manifest: location + hash + order). Never hand-edit or rewrite it. Because the rl-betfair
sources are *in this repo*, the log doubles as a map of which repo docs have been ingested. View:
`wiki_tool.py ingest-log`. Entity-coverage skips are likewise committed source-of-truth — append-only in
`Schema/entity-skips.jsonl` (one `{source_id, term, reason}` per deliberate "not node-worthy" call, via
`entity-skip`); they record a decision, so they version with the notes.

## Dialog record (CoDIAK)
The *semantic* counterpart to the mechanical ingest-log: `Schema/dialog.jsonl` is the append-only
**reasoning trail** — questions, decisions, contradictions + how they were resolved, rejected sources +
why, schema changes. Each entry is addressable and **links to the claim/note/source ids it concerns**,
so *how the wiki got to what it says* is itself recoverable (`dialog trail <id>` / `query dialog-trail`).
Record reasoning as you work (`wiki_tool.py dialog add --kind … --text … --links …`); the skills above
say where. Append-only — never rewrite it.

## Commands you'll use
`init · doctor · register · build · validate · connectivity · search · source-check · ingest-log · log ·
scan · discover · entity-skip · finalize-ingest · stamp-ids · claim-add · claims-lint · relations ·
project · query · dialog · verify · view · coverage · gate · rollback`. Run
`python scripts/wiki_tool.py <cmd> --help`.
- **`coverage`** — substance/provenance extraction-quality gate. **Blocks** `finalize-ingest`
  (strict-by-default; `--no-strict` to override) on three ERRORs: **under-extraction** (substantive
  notes + grounded claims below the source's real size — words/200 for markdown, pages/slides/rows
  for binaries), **claimless** (a substantive `concept`/`entity` note with 0 claims), and
  **unrepresented-entity** (the source NAMES a person/org/tool/place/term that no citing note turns
  into a node/link/alias — the under-extraction size can't see). Also WARNs on page-padding / thin
  stubs. The cure is the `extract` skill — a note per concept/finding, a claim per assertion, a node
  (or `entity-skip`) per named entity; the floor is a *floor, never a target*.
- **`discover` / `entity-skip`** — `discover --source <id>` (or `--path <file>`) prints the
  per-document entity worklist (EXISTING / MISSING / skipped); run it before writing notes. `entity-skip
  --source <id> --term "a,b,c" --reason "…"` records that named candidates are deliberately NOT
  node-worthy (committed to `Schema/entity-skips.jsonl`) so the entity-coverage gate stops flagging
  them. **rl-betfair note:** on dense technical docs `discover` is noisy — it flags acronyms (PPO, EW,
  WIN, LTP), code refs (`L313`, `DiscreteActionShim`), and sentence-initial words. Node/alias the real
  entities; batch-`entity-skip` the rest with one reason. (Engine bug? It's vendored — fix upstream, not
  here.)
- **`gate`** — runs validate + connectivity + claims-lint + coverage and exits 1 on any ERROR, the same
  gate `finalize-ingest` enforces. Also wired as the repo-root `.githooks/pre-commit` so a light or
  entity-incomplete note can't be committed.
- **`rollback`** — undo an ingestion (revert/reset the finalize commit; `--list` to choose,
  `--last`/`--to <hash>`). Note: this wiki shares rl-betfair's git, so each `finalize-ingest` is a real
  rl-betfair commit; `rollback --last` only undoes a commit whose tip is that ingest.
- **Agency** — claims carry `asserted_by` + `verified_by`; `verify` promotes asserted→verified (human
  or cross-source rule); `query unverified-claims` / `flagged-claims` surface unverified LLM claims.
  For rl-betfair this is the cardinal discipline: an *asserted* claim is "candidate to test"; a
  *verified* claim is "held-out-confirmed".
- The **projection** (`project` builds it, `query` reads it) is a derived SQLite lens under
  `.runtime/` — gitignored, regenerable from the files, never the source of truth. `query` answers
  structured questions (claims, provenance, relations, supersession, sources) **with the LLM off**.
- **Views** — `view <name>` renders viewer-independent views (`verified-only`, `unverified-claims`,
  `dialog-trail`, `supersession-timeline`, `draft-vs-stable`) from the projection to `out/` (gitignored)
  — pure projection→template, never the LLM.

## Binding invariants (do not violate)

These are the load-bearing rules for the Engelbart spine — the contract the engine keeps. The first
five are kept **verbatim** from upstream `llm-wiki-v3` and are the ones that must never bend.

1. **Files are the system of record.** The projection DB (Phase D, SQLite) is **derived, regenerable,
   gitignored** — never the truth. Delete it and rebuild from the files at any time.
2. **The zero-dependency core must keep working.** Power-ups (extractors, vectors, MCP, projection,
   Docker) are feature-detected and degrade gracefully. Unzip-and-go with nothing installed still runs
   (BM25 + CLI + skills).
3. **No server-as-store, no Postgres.** Nothing load-bearing depends on Obsidian — it's an optional
   lens only.
4. **Division of labour.** Deterministic code/rules extract *facts & structure* (IDs, dates, entities,
   locators, link graph). The LLM is reserved for *prose synthesis* and relationship *suggestion* — it
   is **never** the authority and never invents provenance.
5. **North star — the dependency-inversion test.** With the LLM switched off, the wiki is still
   structured, addressable, and queryable. If removing the model leaves "just a folder of files," the
   spine isn't done.

The remaining upstream invariants, adapted to this vendored copy:

6. **Works under both Claude Code and Codex.** `AGENTS.md` is canonical; `CLAUDE.md` mirrors it.
7. **Business-tier features** (multi-user, object-level ACL, crypto signing) are **out of scope** unless
   explicitly enabled.
8. **Two runtime surfaces here, one engine.** rl-betfair runs the **standalone CLI** (`wiki_tool.py` +
   skills) and the **read-only MCP server** (`python -m mcp_server`), both over the *same* file record
   and deterministic engine. Every read/query capability must be reachable from the CLI and MCP. (The
   upstream Docker stack is a third surface, not vendored here — see `VENDORED_FROM.md`.)
9. **The engine is vendored, not developed here.** Engine bugfixes/improvements belong upstream in
   `llm-wiki-v3` and are pulled in by re-copy (re-applying the documented subdir patch). The one
   rl-betfair-local change is that patch; keep it minimal. The upstream unit-test suite was not
   vendored — verify a healthy engine with `python scripts/wiki_tool.py doctor` + a `project`/`query`
   round-trip.
