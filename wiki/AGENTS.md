# AGENTS.md ŌĆö operating rules for rl-betfair's in-repo knowledge wiki

You are maintaining **rl-betfair's knowledge wiki** ŌĆö an in-repo copy of the operator's `llm-wiki-v3`
engine (an Engelbart/CoDIAK spine: granular IDs, claim-level provenance, typed links, an embedded
queryable projection, a captured dialog record). Read this file first, every session. The
deterministic engine is `scripts/wiki_tool.py` ŌĆö call it; don't reimplement it. Provenance and exactly
what was copied/dropped from upstream: [`VENDORED_FROM.md`](VENDORED_FROM.md).

> **What this wiki is for.** It is the **deep, pull-on-demand layer** for rl-betfair's accreted
> knowledge ŌĆö the topic-keyed synthesis (claims + provenance, supersession chains, the decision/dialog
> record) deliberately *kept out* of the always-loaded `CLAUDE.md` + `memory/`. A session pulls what it
> needs (the CLI here, or the read-only MCP server registered in `../.mcp.json`) instead of
> carrying it all in resident context. Design rationale: `../plans/memory-improvements/`. The **binding
> invariants** at the bottom of this file govern the engine ŌĆö do not violate them.

## Session start ŌĆö the opener
When the user greets you or sends an **opener trigger** (`open`, `start`, `start wiki`, `go`, `wiki`)
at the beginning of a session, run the opener and reply with a short status block ŌĆö do not wait to be
asked, and do not ingest or edit anything yet:

1. Read this file and the active cloud's `index.md` (`shared/index.md`).
2. Get counts: `python scripts/wiki_tool.py doctor` and `python scripts/wiki_tool.py ingest-log --summary`.
3. Check for pending work in `inbox/pending/` (the `Files/` drop and `urls.md`), `inbox/static/`
   (append-only notes, snapshotted not moved), and the root `Clippings/` folder (web-clipper drops,
   ingested and moved like the inbox).
4. Reply with the **opener**:
   - active cloud + note/source counts,
   - anything pending to ingest (dropped files, unscanned URLs), or "nothing pending",
   - the line: **"What would you like to ingest or ask?"**

Keep it to a few lines. Wait for the user to confirm before ingesting or editing notes.

## The model in brief
- **One cloud: `shared/`.** rl-betfair knowledge is all "work" ŌĆö everything lives in `shared/`
  (`personal/` is unused here). The engine still supports a `personal/` cloud; this wiki simply doesn't
  populate one, and a `shared/` note must never depend on `personal/`.
- **Sources are referenced, not copied.** Register location + hash + title in `Schema/sources.jsonl`;
  never copy source bytes into the repo. For the repo-scan path the source IS the in-repo file (e.g.
  `plans/.../findings.md`) ŌĆö referenced in place. Dropped inbox files are the one staging exception;
  `inbox/static/` notes are referenced in place and snapshotted, never moved.
- **The graph is the product.** Every note links into a hub (`shared/index.md`, auto-generated by
  `build`); isolated notes and unreachable clusters are failures, not cosmetic.

## Ingestion ŌĆö two entry points
rl-betfair knowledge enters the wiki two ways:

1. **Repo-scan (bulk ŌĆö the rl-betfair corpus).** `python scripts/scan_repo.py` walks the rl-betfair
   tree (`plans/**/*.md`, `docs/**/*.md`, and root knowledge docs: `CLAUDE.md`, `genes_census.md`,
   `plans/EXPERIMENTS.md`, `plans/EXPLORATIONS.md`), **registers each as a source** (referenced in
   place ŌĆö the files stay in the repo, never copied), and queues them for the **`batch`** skill. Then
   work the queue one source at a time. Scope is markdown knowledge; ingesting `.py` code is a later
   option. `scan_repo.py` is rl-betfair's own thin wrapper over the engine's `register` + `batch` ŌĆö
   see `python scripts/scan_repo.py --help`.
2. **Inbox drop (external ML docs you find).** The inbox is for **outside** material ŌĆö an ML paper, a
   blog post, a vendor doc. Drop a file into `inbox/pending/Files/`, a URL into `inbox/pending/urls.md`,
   or a web-clip into `Clippings/`, then ingest via the `ingest` skill; `finalize-ingest` moves it to
   `processed/`. This is the engine as shipped.

Both paths produce the **same note shapes:** a `concept` note per mechanism (the matcher, equal-profit
sizing, naked variance, the held-out protocol), a `synthesis` note per cross-cohort lesson, a `project`
note per campaign, `claims` carrying the numbers + their source run, `dialog` entries for the decisions.

## Ingesting a source (one source at a time)
1. **Register** the source (`wiki_tool.py register ŌĆ”`) ŌåÆ get a `src-id`.
2. **Read it fully.** Never skim or sample. For large/dense docs, use the extraction engine below.
3. **Search first:** `wiki_tool.py search "ŌĆ”"` across the active cloud before creating notes.
4. **Write breadth.** A new Entity for every named cohort/campaign/mechanism/tool; a new Concept for
   every distinct idea; Topics/Projects where warranted. Typical ingest touches many notes.
5. **Lift key quotes into the notes** so citations survive even if the source later moves.
6. **Cross-link** every new note to at least one hub and to related notes via `[[wiki-links]]`.
7. **Finalize:** `wiki_tool.py finalize-ingest`. The ingest is not done until this passes.

## Static (append-only) sources
A note you keep reopening and appending to lives in `inbox/static/`. Unlike a `pending/` drop, a static
source is **referenced in place and never moved** ŌĆö you keep editing the same file. To ingest one (see
the `static` skill):
1. **Register** it if new (`register --path inbox/static/<f>`); the src-id is stable across edits.
2. **Diff, don't re-read everything.** Compare it to the latest snapshot in `inbox/processed/static/`
   and ingest **only the newly-added content**. No prior snapshot ŌćÆ ingest it all.
3. **Finalize.** `finalize-ingest` copies a fresh dated snapshot to `inbox/processed/static/` (only
   when the file changed) and leaves your original in place for the next entry.

## Web clippings
The Obsidian Web Clipper saves articles to the root `Clippings/` folder. Treat each as a normal pending
drop: ingest it like a `Files/` drop, then `finalize-ingest` moves it to `inbox/processed/Clippings/`.

## The anti-shortcut extraction engine (large / dense docs)
Producing 5 notes from a 60-page campaign writeup is a failure. For such docs:
1. **Segment** by structure: doc ŌåÆ section ŌåÆ finding ŌåÆ claim (provenance kept).
2. **Process each segment independently** ŌĆö enumerate *every* entity/concept/finding/decision in that
   segment before moving on. Don't read the whole doc then emit a handful of notes.
3. A segment that yields **zero** notes must be justified (genuinely contentless) or it's flagged.
4. **Dedup/merge** concepts that recur across segments ŌĆö link, don't duplicate.
5. Expect to clear the **coverage floor**. `finalize` warns when you're under it ŌĆö treat that warning
   as "what did I skip?"
Granularity: structural-segment by default; finer for reference-grade material.

## Querying
Read `index.md` ŌåÆ `search` ŌåÆ open top notes ŌåÆ answer from compiled notes with citations. Open a raw
source only if a note is a seed/incomplete or the user wants verification. **File worthwhile answers
back** as a `query` note so knowledge compounds.

## Tags & types
One structural **type** per note (`topic, concept, entity[+subtype], project, synthesis, log, query`)
plus controlled **context tags** (`work, home, research, meetings, lessons`). Unknown tags fail
`validate`. (rl-betfair knowledge is `work` / `research` / `lessons`; `home` is unused.)

## Hard rules
- Don't copy source bytes into the repo (register references instead).
- Don't invent citations. No source ŌćÆ find one or don't make the claim.
- Don't leave orphans; every note reaches a hub.
- Don't commit secrets or machine-local paths (the `audit_public` gate blocks them).
- When a rule itself is wrong (linking/ingest keeps going astray), fix the rule here and in the
  schema/tooling ŌĆö not just the symptom.
- **"Process everything" / "ingest everything" means *finish the job*.** Don't mark heavy sources
  `batch done` with only a structural-overview skim and call the rest "TBD" ŌĆö that's the exact shortcut
  the extraction engine forbids. Clear the coverage floor before closing the source. Don't ask which to
  prioritise ŌĆö process all of them.

## Replication
Every `register` and `finalize-ingest` is recorded append-only in `Schema/ingest-log.jsonl` (the
replication manifest: location + hash + order). Never hand-edit or rewrite it. Because the rl-betfair
sources are *in this repo*, the log doubles as a map of which repo docs have been ingested. View:
`wiki_tool.py ingest-log`.

## Dialog record (CoDIAK)
The *semantic* counterpart to the mechanical ingest-log: `Schema/dialog.jsonl` is the append-only
**reasoning trail** ŌĆö questions, decisions, contradictions + how they were resolved, rejected sources +
why, schema changes. Each entry is addressable and **links to the claim/note/source ids it concerns**,
so *how the wiki got to what it says* is itself recoverable (`dialog trail <id>` / `query dialog-trail`).
Record reasoning as you work (`wiki_tool.py dialog add --kind ŌĆ” --text ŌĆ” --links ŌĆ”`); the skills above
say where. Append-only ŌĆö never rewrite it.

## Commands you'll use
`init ┬Ę doctor ┬Ę register ┬Ę build ┬Ę validate ┬Ę connectivity ┬Ę search ┬Ę source-check ┬Ę ingest-log ┬Ę log ┬Ę
scan ┬Ę finalize-ingest ┬Ę stamp-ids ┬Ę claim-add ┬Ę claims-lint ┬Ę relations ┬Ę project ┬Ę query ┬Ę dialog ┬Ę
verify ┬Ę view ┬Ę coverage ┬Ę rollback`. Run `python scripts/wiki_tool.py <cmd> --help`.
- **`coverage`** ŌĆö substance-aware extraction-quality gate (flags page-padding / thin stubs; the floor
  is a *floor, never a target* ŌĆö extract per concept/finding, never per page).
- **`rollback`** ŌĆö undo an ingestion (revert/reset the finalize commit; `--list` to choose,
  `--last`/`--to <hash>`). Note: this wiki shares rl-betfair's git, so each `finalize-ingest` is a real
  rl-betfair commit; `rollback --last` only undoes a commit whose tip is that ingest.
- **Agency** ŌĆö claims carry `asserted_by` + `verified_by`; `verify` promotes assertedŌåÆverified (human
  or cross-source rule); `query unverified-claims` / `flagged-claims` surface unverified LLM claims.
  For rl-betfair this is the cardinal discipline: an *asserted* claim is "candidate to test"; a
  *verified* claim is "held-out-confirmed".
- The **projection** (`project` builds it, `query` reads it) is a derived SQLite lens under
  `.runtime/` ŌĆö gitignored, regenerable from the files, never the source of truth. `query` answers
  structured questions (claims, provenance, relations, supersession, sources) **with the LLM off**.
- **Views** ŌĆö `view <name>` renders viewer-independent views (`verified-only`, `unverified-claims`,
  `dialog-trail`, `supersession-timeline`, `draft-vs-stable`) from the projection to `out/` (gitignored)
  ŌĆö pure projectionŌåÆtemplate, never the LLM.

## Binding invariants (do not violate)

These are the load-bearing rules for the Engelbart spine ŌĆö the contract the engine keeps. The first
five are kept **verbatim** from upstream `llm-wiki-v3` and are the ones that must never bend.

1. **Files are the system of record.** The projection DB (Phase D, SQLite) is **derived, regenerable,
   gitignored** ŌĆö never the truth. Delete it and rebuild from the files at any time.
2. **The zero-dependency core must keep working.** Power-ups (extractors, vectors, MCP, projection,
   Docker) are feature-detected and degrade gracefully. Unzip-and-go with nothing installed still runs
   (BM25 + CLI + skills).
3. **No server-as-store, no Postgres.** Nothing load-bearing depends on Obsidian ŌĆö it's an optional
   lens only.
4. **Division of labour.** Deterministic code/rules extract *facts & structure* (IDs, dates, entities,
   locators, link graph). The LLM is reserved for *prose synthesis* and relationship *suggestion* ŌĆö it
   is **never** the authority and never invents provenance.
5. **North star ŌĆö the dependency-inversion test.** With the LLM switched off, the wiki is still
   structured, addressable, and queryable. If removing the model leaves "just a folder of files," the
   spine isn't done.

The remaining upstream invariants, adapted to this vendored copy:

6. **Works under both Claude Code and Codex.** `AGENTS.md` is canonical; `CLAUDE.md` mirrors it.
7. **Business-tier features** (multi-user, object-level ACL, crypto signing) are **out of scope** unless
   explicitly enabled.
8. **Two runtime surfaces here, one engine.** rl-betfair runs the **standalone CLI** (`wiki_tool.py` +
   skills) and the **read-only MCP server** (`python -m mcp_server`), both over the *same* file record
   and deterministic engine. Every read/query capability must be reachable from the CLI and MCP. (The
   upstream Docker stack is a third surface, not vendored here ŌĆö see `VENDORED_FROM.md`.)
9. **The engine is vendored, not developed here.** Engine bugfixes/improvements belong upstream in
   `llm-wiki-v3` and are pulled in by re-copy (re-applying the documented subdir patch). The one
   rl-betfair-local change is that patch; keep it minimal. The upstream unit-test suite was not
   vendored ŌĆö verify a healthy engine with `python scripts/wiki_tool.py doctor` + a `project`/`query`
   round-trip.
